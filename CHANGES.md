## Item Variant System
**Date:** 2026-03-26

### Migration File
**`item_variant_migration.sql`** — run this once against `craftcottage_bwn_db` before deploying any of the three apps.

```sql
-- Creates item_variant table
-- Alters order_item (adds sku_id, variant_label; widens PK)
```

#### Step 1 — New table `item_variant`
| Column | Type | Notes |
|---|---|---|
| `sku_id` | `VARCHAR(50)` PK | Auto-generated by vendor app (`UniqueId::generateId('sku', 12)`) |
| `item_id` | `VARCHAR(10)` FK → `item.item_id` | Parent product |
| `size` | `VARCHAR(30)` nullable | e.g. `XL`, `M`, `42` |
| `color` | `VARCHAR(30)` nullable | e.g. `Red`, `Blue` |
| `unit_mrp` | `FLOAT UNSIGNED` | Variant MRP |
| `unit_price` | `FLOAT UNSIGNED` | Variant selling price |
| `stock` | `SMALLINT UNSIGNED` default `0` | Variant stock count |
| `active` | `TINYINT(1)` default `1` | `0` hides variant from customers |
| `created_on` | `TIMESTAMP` | Auto |
| `modified_on` | `TIMESTAMP` | Auto-update |

#### Step 2 — Alter `order_item`
| Change | Detail |
|---|---|
| Add `sku_id VARCHAR(50) NOT NULL DEFAULT ''` | `''` for non-variant orders |
| Add `variant_label VARCHAR(100) NOT NULL DEFAULT ''` | Snapshot of variant at order time, e.g. `"XL / Red"` |
| Drop old PK `(order_id, item_id)` | |
| Add new PK `(order_id, item_id, sku_id)` | Allows same item in two different variants in one order |

### Deployment Order
1. Run `item_variant_migration.sql` on the database
2. Deploy `ecraft-vendor-phpcodeignitor4` (vendors can now create variants)
3. Deploy `ecraft-client-nextjs` (customers see variant UI)
4. Deploy `ecraft-admin-php-codeignitor4` (admin sees variant info in orders)

### Backward Compatibility Summary
- Items with no variants in `item_variant`: client shows no variant UI, behaves as before
- All historical `order_item` rows: `sku_id = ''`, `variant_label = ''` — no data to migrate
- All existing vendor/admin PHP queries: use `$sku_id = ''` default — no breakage
